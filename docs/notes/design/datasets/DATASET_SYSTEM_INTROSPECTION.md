## Data Source Tracking

The loader now tracks where data comes from via `DataSource` enum:
- `S3Cache` - Retrieved from S3 cache bucket
- `LocalCache` - Loaded from local disk cache
- `OriginalUrl` - Downloaded from original source
- `Skipped` - Dataset unavailable
- `Embedded` - Built into binary (test fixtures)

## Baseline Metrics

The registry now includes expected F1 scores from published papers:
- `DatasetId::expected_f1()` - Fine-tuned model baseline
- `DatasetId::expected_zero_shot_f1()` - Zero-shot estimate (~10% lower)

## Architectural Status

### 1. DatasetId Enum Synchronization ✅
Both `DatasetId` enums are now synchronized:
- `anno/src/eval/dataset_registry.rs` → 451 variants (generated by macro)
- `anno/src/eval/loader.rs` → 451 variants (now complete)

**Status**: COMPLETE - All 451 datasets have loader implementations.

The `scripts/check_dataset_sync.py` script verifies parity and is integrated into CI.

### 2. Loading Abstraction
Each dataset has different:
- Formats (CoNLL, JSONL, TSV, XML, Custom)
- Schemas (entity types, annotation schemes)
- Access methods (direct URL, HuggingFace API, gated/licensed)

The `loader.rs` now has comprehensive format-specific parsers:
- ✅ CoNLL/BIO format
- ✅ JSONL format (HuggingFace style)
- ✅ DocRED JSON format (relation extraction)
- ✅ HuggingFace datasets-server API
- ✅ Custom parsers for specialized datasets

Remaining considerations:
- No streaming support for very large datasets (defer to future)
- Some specialized formats (BRAT, Standoff, XML) fall back to generic parsers
