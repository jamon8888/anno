# lancedb 0.29 Workspace Bump — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bump the workspace `lancedb` pin from `=0.27.2` to `=0.29.x` (latest stable on the 0.29 line at execution time), fix any compile breaks in `crates/anno-rag/src/store.rs`, and verify the v0.5 retrieval baseline holds. This unblocks anno-rag v0.6 (native FTS) and anno-memory v0.1 (native FTS + scalar indexes + `RRFReranker`).

**Architecture:** Single workspace pin change in the root `Cargo.toml`. `lancedb` is consumed in exactly one place — `crates/anno-rag/src/store.rs` — and uses a small surface (`connect`, `Table`, `ExecutableQuery`/`QueryBase`, `nearest_to`, `count_rows`, `list_indices`, `create_index`, `Index::IvfHnswSq` + `IvfHnswSqIndexBuilder`). Most likely zero or one signature drift. The plan captures the v0.5 numerical baseline before the bump and asserts it after.

**Tech Stack:** Rust 2021, `lancedb 0.29.x`, `cargo` workspace.

---

## File Structure

- **Modify** `Cargo.toml` (workspace root) — `lancedb = "=0.27.2"` → `lancedb = "=0.29.x"`.
- **Regenerate** `Cargo.lock` — `cargo update -p lancedb`.
- **Modify (only if needed)** `crates/anno-rag/src/store.rs` — fix any 0.29 API drift.
- **Modify (only if needed)** `crates/anno-rag/src/error.rs` — fix any 0.29 error-type drift (`lancedb::Error` is `pub use`'d here).

No new files. No new dependencies. No new tests written by this plan (v0.5 has tests; we run them against the new pin).


Expected: either green or a small number of errors localized to `anno-rag/src/store.rs`. The high-risk surfaces are:
- `Index::IvfHnswSq(IvfHnswSqIndexBuilder::default())` — builder constructor may have moved or its default may have new required fields.
- `Table::create_index(&["vector"], Index::...)` returning an `IndexBuilder` whose `.execute()` chain may have changed.
- `lancedb::Error` variants may have been renamed (e.g. new error categories split out).
- `QueryBase::nearest_to` may now require `Vec<f32>` vs `&[f32]` or vice versa.

- [ ] **Step 2: For each compile error, fix it minimally**

For each error reported by Step 1, apply the smallest possible change. Do NOT refactor. Do NOT add features. Examples of expected fixes (these are illustrative — apply only what the compiler asks for):


## Task 4: Run the tests and bench, verify baseline

**Files:**
- Read: `crates/anno-rag/tests/*`, `crates/anno-rag/benches/bench_recall.rs`.

- [ ] **Step 1: Run unit + integration tests for anno-rag**

Run:
```powershell
cargo test -p anno-rag --all-features 2>&1 | Tee-Object -FilePath test_0.29.txt
```

Expected: green. If any tests fail, the failure mode is one of:
- API drift the build phase let through (e.g. a panic on a now-renamed enum variant) — go back to Task 3.
- Numerical drift in retrieval (e.g. an internal default changed in lancedb's IVF index) — proceed to Step 2 to quantify.

- [ ] **Step 2: Run the retrieval bench**

Run:
```powershell
cargo bench -p anno-rag --bench bench_recall 2>&1 | Tee-Object -FilePath bench_0.29.txt
```

Expected: bench completes. Extract `recall@10`, `nDCG@10`, and search-latency numbers as in Task 1 Step 2.

- [ ] **Step 3: Compare against `BASELINE.txt`**

Manually diff. Acceptance criteria:
- `recall@10` within ±0.02 of the 0.27.2 baseline.
- `nDCG@10` within ±0.02 of the 0.27.2 baseline (if reported).
- Search p95 latency within +20% of 0.27.2 baseline.
- Ingest p95 latency within +30% of 0.27.2 baseline (LanceDB has shipped multiple ingest perf changes in this range; some movement is expected).

If any number is outside its band:
- For retrieval-quality drift, **stop and investigate** — do not merge. The likely cause is an index-default change. Look for new fields in `IvfHnswSqIndexBuilder` and pin the old defaults explicitly.
- For latency drift, log a follow-up issue; do not block the bump.

- [ ] **Step 4: Confirm peak RSS still under 1.5 GB**

Run the v0.5 perf-budget check (replace path if the script lives elsewhere):
```powershell
.\scripts\check_peak_rss.ps1 -Crate anno-rag -BudgetMb 1500
```
If no such script exists yet, run the bench with `cargo bench` under a memory profiler (`heaptrack`, `valgrind --tool=massif`, or Windows `Get-Process | Select-Object PeakWorkingSet64`) and visually confirm.

Expected: under 1.5 GB. If breached, do not merge — file an issue and downgrade pin.
