# Anno Installer Phase 2A — cargo-dist Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the Phase 1 hand-rolled `.zip`/`.tar.gz` release pipeline with a cargo-dist–managed workflow that produces real native installers (`.msi` on Windows, `.dmg` on macOS, `AppImage`+`.deb` on Linux) plus shell/PowerShell install scripts and a Homebrew tap formula, covering all four targets including Linux which was absent from Phase 1.

**Architecture:** Add `[workspace.metadata.dist]` config to `Cargo.toml`, mark distributed vs. non-distributed crates with `[package.metadata.dist]`, run `cargo dist generate` to produce `.github/workflows/release.yml`, then surgically patch the generated file to preserve two anno-specific requirements: the MSVC CRT dynamic-link env vars on Windows and the gateway boot-smoke test. The old `release-binaries.yml` is deleted.

**Tech Stack:** cargo-dist 0.32.0, GitHub Actions, WiX v4 (`.msi`), create-dmg (`.dmg`), AppImage tooling (Linux).

### Task 3: Patch MSVC CRT alignment

**Files:**
- Modify: `.github/workflows/release.yml`

The `ort` crate (ONNX Runtime) links with the dynamic MSVC CRT (`/MD`). By default, `esaxx-rs` (pulled in via `tokenizers`) compiles its C++ with the static CRT (`/MT`), causing a `LNK2038 RuntimeLibrary mismatch` on Windows. The Phase 1 workflow had an explicit step to fix this; it must be re-injected into the cargo-dist workflow.

- [ ] **Step 1: Find the `cargo dist build` line in the generated workflow**

```bash
grep -n "cargo dist build" .github/workflows/release.yml
```

Expected: one or more lines like:

```
42:      run: cargo dist build --artifacts=local ...
```

Note the line number (call it LINE_N).

- [ ] **Step 2: Insert the MSVC CRT step immediately before the `cargo dist build` step**

Open `.github/workflows/release.yml`. Locate the step containing `cargo dist build`. Insert the following YAML block **before** that step. The indentation must match the surrounding steps (typically 6 spaces):

```yaml
      # ANNO-PATCH: MSVC CRT alignment — re-apply after every `cargo dist generate`.
      # ort (ONNX Runtime) links /MD; esaxx-rs (via tokenizers) defaults to /MT.
      # Forcing -crt-static=off + -MD aligns both to the dynamic CRT and avoids
      # LNK2038 RuntimeLibrary mismatch on Windows.
      # See docs/superpowers/plans/2026-05-04-gliner2-fastino-finalization.md F1.2.
      - name: Align MSVC CRT (Windows)
        if: runner.os == 'Windows'
        shell: bash
        run: |
          echo "RUSTFLAGS=-C target-feature=-crt-static" >> "$GITHUB_ENV"
          echo "CFLAGS_x86_64-pc-windows-msvc=-MD" >> "$GITHUB_ENV"
          echo "CXXFLAGS_x86_64-pc-windows-msvc=-MD" >> "$GITHUB_ENV"
```

- [ ] **Step 3: Verify the step order**

```bash
grep -n "Align MSVC CRT\|cargo dist build" .github/workflows/release.yml
```

Expected: `Align MSVC CRT` appears **before** `cargo dist build` on the line number axis.

- [ ] **Step 4: Validate YAML syntax**

```bash
python -c "import yaml, pathlib; yaml.safe_load(pathlib.Path('.github/workflows/release.yml').read_text()); print('YAML valid')"
```

Expected: `YAML valid`

- [ ] **Step 5: Commit**

```bash

git commit -m "ci: patch release workflow — MSVC CRT alignment (ANNO-PATCH)"
```

### Task 4: Patch gateway boot smoke

**Files:**
- Modify: `.github/workflows/release.yml`

`anno-privacy-gateway` is a long-running HTTP server. Its `--help` flag starts the server and blocks (it does not print usage and exit). The Phase 1 workflow used dedicated boot-smoke scripts (`scripts/release/smoke-gateway.ps1` and `scripts/release/smoke-gateway.sh`) that start the server on `ANNO_GATEWAY_LISTEN=127.0.0.1:0`, wait 3 seconds, verify it is still alive, and then terminate it. These scripts must be called in the cargo-dist workflow after the binary is built.

- [ ] **Step 1: Find the `cargo dist build` step and the next `upload-artifact` step**

```bash
grep -n "cargo dist build\|upload-artifact\|Upload artifact" .github/workflows/release.yml | head -10
```

Expected: `cargo dist build` appears before the first `upload-artifact` reference. Note both line numbers.

- [ ] **Step 2: Insert gateway smoke steps between build and upload**

Open `.github/workflows/release.yml`. Locate the `cargo dist build` step. Insert the following two steps **after** that step and **before** the `upload-artifact` step. Indentation must match (typically 6 spaces):

```yaml
      # ANNO-PATCH: Gateway boot smoke — re-apply after every `cargo dist generate`.
      # anno-privacy-gateway is a server binary; --help starts the server (does not print and exit).
      # The smoke scripts start on ANNO_GATEWAY_LISTEN=127.0.0.1:0, wait 3s, verify alive, terminate.
      - name: Gateway boot smoke (Windows)
        if: runner.os == 'Windows'
        shell: pwsh
        run: .\scripts\release\smoke-gateway.ps1 -BinaryPath ".\target\${{ join(matrix.targets, ' ') }}\release\anno-privacy-gateway.exe"

      - name: Gateway boot smoke (Unix)
        if: runner.os != 'Windows'
        shell: bash
        run: ./scripts/release/smoke-gateway.sh "./target/${{ join(matrix.targets, ' ') }}/release/anno-privacy-gateway"
```

Note: `join(matrix.targets, ' ')` evaluates to the single target triple for the current matrix row (e.g. `x86_64-pc-windows-msvc`). cargo-dist sets `matrix.targets` as a list; the first and only element is the build target.

- [ ] **Step 3: Verify the step order**

```bash
grep -n "cargo dist build\|Gateway boot smoke\|upload-artifact\|Upload artifact" .github/workflows/release.yml | head -15
```

Expected: line numbers in order: `cargo dist build` → `Gateway boot smoke (Windows)` → `Gateway boot smoke (Unix)` → `upload-artifact`.

- [ ] **Step 4: Validate YAML syntax**

```bash
python -c "import yaml, pathlib; yaml.safe_load(pathlib.Path('.github/workflows/release.yml').read_text()); print('YAML valid')"
```

Expected: `YAML valid`
