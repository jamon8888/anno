# anno-rag v0.7 — Anonymization Eval Harness + Email Detection

**Status:** Design — approved, pending spec review
**Date:** 2026-05-15
**Scope:** anno-rag crate only

## 1. Motivation

An investigation of the v0.5/v0.6 anonymization pipeline found that the
PII-detection layer is **unmeasured**. The regex layer (NIR, SIRET, IBAN-FR,
French phone) is solid and correct-by-construction. The NER layer
(`GLiNER2Fastino` multi-v1, types person/organization/location) is plausible
but:

- has **no CI test coverage** — every `detect.rs` and `e2e.rs` test is
  `#[ignore]`d (needs the model cache);
- has **no precision/recall measurement** — there is no labelled PII test set,
  so "does anonymization work well on legal text?" cannot be answered with a
  number;
- has **confirmed gaps** — email addresses are not detected at all (the
  `email_entity` helper in `vault.rs` is test-only; there is no email regex
  and no email NER type).

This is the same failure mode v0.6 fixed for retrieval: you cannot improve
what you cannot measure. v0.7 builds the measuring stick for anonymization
and closes the highest-impact gap (email).

## 2. Scope

**In scope:**
- An annotated synthetic PII corpus (~35 French legal documents + a
  `pii_annotations.toml` ground-truth file).
- A PII evaluation harness: per-category precision/recall/F1 with
  overlap-span matching.
- A two-tier CI gate: a model-free regex-category gate that runs on every CI
  build, and a model-requiring NER-category gate that runs in the bench job.
  Both hard-gate on **recall** (a missed PII is a privacy leak).
- Email detection: a regex added to the `detect.rs` pattern pack.

**Out of scope (deferred to v0.8+):**
- Street-level postal addresses, RCS / TVA / CNI / passport numbers, dates of
  birth, court docket numbers (numéro de rôle / RG).
- Fixing the known byte/char offset dedup bug on accented text — v0.7
  *measures* its effect (via precision) but does not fix it.
- NER confidence-threshold tuning; a legal-tuned NER model.
- A real PII corpus slot (`ANNO_RAG_PII_CORPUS`).

## 3. Architecture

### Component A — Annotated PII corpus

Location: `crates/anno-rag/tests/fixtures/pii_corpus/`

- ~35 French legal `.txt` documents across families (contrats de prestation,
  CDI, mises en demeure, baux, statuts — ~7 per family).
- PII is **dense** and **unique-valued per document**: within any one
  document, every PII string is unique (one `FR76 …` IBAN, one `Marie
  Dupont`, etc.). This is the key constraint — it lets the harness resolve a
  span by string search without ambiguity.
- Covers all eight categories: `NIR`, `SIRET`, `IBAN_FR`, `PhoneNumber`,
  `Email`, `Person`, `Organization`, `Location`.
- Deliberately includes **traps**: legal terms that resemble entities but are
  not (« le Prestataire », « la Société » with no proper noun, « Tribunal de
  commerce ») — to measure precision (over-scrubbing).

### Component B — PII eval harness

Location: `crates/anno-rag/src/pii_eval.rs` — a new module, distinct from
`eval.rs` (PII detection is a different concern from retrieval).

- `pii_annotations.toml` loader + a consistency checker.
- Pure metric functions: per-category precision / recall / F1, with
  overlap-span matching. Unit-testable without a model.
- `score_detections(detected, truth) -> PiiScores`.

### Component C — Email detection

Location: `crates/anno-rag/src/detect.rs`

- A new `email` regex in `FrPatterns`, emitted as
  `EntityCategory::Email` / `DetectionSource::Pattern`.
- A refactor: extract `detect_patterns(text) -> Vec<DetectedEntity>` — the
  whole regex layer, **free function, no `self`, no NER** — so the model-free
  tier can call it. `Detector::detect` becomes `detect_patterns(text)` + NER
  entities, then the existing merge/dedup.

### Component D — model-free regex tier

Location: `crates/anno-rag/tests/pii_regex.rs`

- An integration test that runs `detect_patterns` over the annotated corpus,
  computes per-category precision/recall/F1 for the five regex categories
  (`NIR`, `SIRET`, `IBAN_FR`, `PhoneNumber`, `Email`), and **hard-gates on
  recall** against `pii_baseline.toml`. Runs on **every CI build** — fast, no
  model.

### Component E — model-requiring NER tier

Location: `crates/anno-rag/benches/bench_pii.rs`

- A criterion bench that runs the full `Detector::detect` over the annotated
  corpus, computes per-category precision/recall/F1 for the three NER
  categories (`Person`, `Organization`, `Location`), and emits
  `PII_RECALL_<category>=<float>` lines on stderr for CI scraping. Gated in
  `bench.yml`. Runs in the bench job.

**Constraint:** no real-corpus slot in v0.7 — only the generated fixtures.

## 4. Annotated corpus and annotation format

### Documents

~35 French legal `.txt` files under `tests/fixtures/pii_corpus/`, ~7 per
family (prestation, cdi, miseendemeure, bail, statuts). Realistic French
legal register. PII is dense and **unique-valued within each document**.

### `pii_annotations.toml`

```toml
[[doc]]
file = "prestation_01.txt"
pii = [
  { category = "Organization", text = "ATELIER NUMÉRIQUE" },
  { category = "Person",       text = "Claire Fontaine" },
  { category = "SIRET",        text = "81234567800019" },
  { category = "IBAN_FR",      text = "FR76 3000 4000 0500 0612 3456 789" },
  { category = "PhoneNumber",  text = "06 12 34 56 78" },
  { category = "Email",        text = "contact@atelier-numerique.fr" },
]
```

`category` values map to `EntityCategory`: the customs `NIR` / `SIRET` /
`IBAN_FR` as strings, the natives `Person` / `Organization` / `Location` /
`Email` / `PhoneNumber`.

### Consistency check

A unit test asserts that, for every `{file, pii}` entry, each `text` appears
**exactly once** in the file. It fails if an annotated string is absent or
ambiguous (multiple occurrences). This guarantees offset resolution by
search is unambiguous.

### Offset resolution

At eval time the harness does `doc_content.find(text)` →
`(start, start + text.len())` = the ground-truth span for that category.

## 5. Metrics and matching

Location: `crates/anno-rag/src/pii_eval.rs` — pure functions, unit-tested on
toy inputs (no model).

### Span matching — overlap

A detected span `(ds, de, cat)` **matches** a true span `(ts, te, cat)` when
the categories are equal **and** the half-open intervals overlap
(`ds < te && ts < de`). Overlap, not exact equality, absorbs the known
accent-offset dedup bug — an IBAN detected one character off still counts as
scrubbed.

### Per-category counting

For each category independently:
- **TP**: a true span with ≥ 1 overlapping detected span.
- **FN**: a true span with no overlapping detected span → a **PII leak**.
- **FP**: a detected span overlapping no true span → over-scrubbing.
- Matching is greedy left-to-right; a detected span can be credited to at
  most one true span, so one large detected span cannot inflate TP.

### Metrics

- `precision = TP / (TP + FP)`, `recall = TP / (TP + FN)`,
  `f1 = 2·P·R / (P + R)`.
- Vacuous case: a category with no true spans and no detected spans scores
  P/R/F1 = 1.0.
- `PiiScores { per_category: HashMap<String, CategoryScore> }` where
  `CategoryScore { precision, recall, f1, tp, fp, fn_ }`.

### `pii_baseline.toml`

Committed per-category recall baselines:

```toml
[recall]
NIR = 1.0
SIRET = 1.0
IBAN_FR = 1.0
PhoneNumber = 0.95
Email = 0.98
Person = 0.0          # filled from bench_pii during the final build phase
Organization = 0.0
Location = 0.0
```

## 6. CI gates

- **Model-free tier** (`pii_regex.rs`, every CI build): for `NIR`, `SIRET`,
  `IBAN_FR`, `PhoneNumber`, `Email` — `recall ≥ baseline × 0.98`. A failure
  is a regex-layer leak. Precision is **reported but non-blocking** (wrongly
  scrubbing a legal term is not a privacy leak).
- **Bench tier** (`bench_pii.rs` → `bench.yml`, bench job): the same
  recall-only gate for `Person`, `Organization`, `Location`.

## 7. Email detection

Location: `crates/anno-rag/src/detect.rs`

- A new `email: Regex` field in `FrPatterns`, with a pragmatic
  RFC-5321-ish pattern:
  `[A-Za-z0-9._%+\-]+@[A-Za-z0-9.\-]+\.[A-Za-z]{2,}`. This covers
  contract-style email addresses; quoted-local-part and other exotic forms
  are intentionally not handled (YAGNI).
- Emitted as
  `DetectedEntity { category: EntityCategory::Email, source: DetectionSource::Pattern, confidence: 1.0 }`.
- Dedup priority: `Pattern` = 0, identical to NIR/SIRET/IBAN/phone — no
  change to the existing sort/dedup logic.

## 8. `detect.rs` refactor

To support the model-free tier:

- Extract `detect_patterns(text: &str) -> Vec<DetectedEntity>` — the entire
  regex layer (NIR, SIRET, IBAN, phone, email), as a free function with no
  `self` and no NER dependency.
- `Detector::detect` becomes: `detect_patterns(text)` + NER entities → merge
  → dedup. The sort/dedup logic is unchanged.
- `detect_patterns` is made `pub` so `pii_regex.rs` can call it without
  loading the model. Minor public-API addition; anno-rag is pre-1.0.

## 9. Testing

- `pii_eval.rs`: unit tests for the metric functions (precision/recall/F1,
  overlap matching, vacuous case) on toy inputs with known expected values —
  no model.
- `pii_eval.rs`: a corpus consistency test — every annotated `text` is
  findable exactly once in its document.
- `detect.rs`: a test that the email regex matches contract-style emails and
  **rejects** non-emails (a bare `@`, `a@b` with no TLD).
- `pii_regex.rs`: the model-free eval is itself an integration test plus the
  recall gate.
- `bench_pii.rs`: the NER eval is the model-requiring integration test.
