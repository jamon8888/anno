# anno-rag v0.6 — Hybrid Retrieval + Legal Eval Harness

**Status:** Design — approved, pending spec review
**Date:** 2026-05-14
**Scope:** anno-rag crate only

## 1. Motivation

A RAG-quality investigation of the v0.5 pipeline surfaced two blocking gaps:

1. **No real evaluation.** The v0.5 `bench_recall` runs on 5 fixture documents
   with k=10 — every document is always returned, so recall saturates at 1.0
   and measures nothing. You cannot improve what you cannot measure.
2. **Vector-only retrieval.** `store.rs` does dense KNN only (`nearest_to`).
   Legal French text is saturated with exact-match terms — article numbers
   ("Article L. 1234-5"), statute references, defined terms ("le Prestataire"),
   amounts, dates, IBAN/SIRET. Dense embeddings are structurally weak at exact
   lexical match. A secondary bug compounds this: the e5 embedder applies the
   `"passage: "` prefix to **queries** as well as passages, violating the e5
   asymmetric-prefix convention and measurably degrading retrieval.

v0.6 builds the measuring stick **and** the first improvement that stick
validates, in one coherent spec.

## 2. Scope

**In scope (P0 + eval):**
- Legal evaluation harness: synthetic corpus + graded queries + recall@k /
  nDCG@10 metrics + CI gate.
- e5 query/passage prefix fix.
- Native LanceDB hybrid search: FTS (Tantivy) index + `RRFReranker`.

**Out of scope (deferred to v0.7+):**
- Legal-structure-aware chunking (article/clause anchoring).
- Pseudonymization-vs-embedding rework (stable entity tokens / embedding
  original text).
- Cross-encoder / ColBERT rerankers.
- Metadata prefiltering (`folder_path`, `doc_id`).
- Larger / legal-domain embedding model.
- Contextual chunk enrichment.

**Carried constraint:** the v0.5 peak-RSS cap of **< 1.5 GB** still holds. The
Tantivy FTS index is disk-backed; its resident memory cost is small.

## 3. Architecture

Two coupled deliverables in one spec.

### Flow

```
ingest (unchanged)            search
  └─ + build FTS index          requête
                                  └─ detect → pseudonymize
                                  └─ embed_query(pseudo_q)   → query vector
                                  └─ pseudo_q text           → FTS term
                                  └─ LanceDB hybrid (vector + FTS)
                                  └─ RRFReranker
                                  └─ top-k SearchHit
```

### Component A — Legal eval harness

Location: `crates/anno-rag/`

- `tests/fixtures/eval_corpus/` — ~60 synthetic French legal `.txt` documents,
  versioned in the repo.
- `tests/fixtures/eval_corpus/queries.toml` — ~40 concept-oriented queries,
  each with a list of relevant documents and a graded relevance score.
- `src/eval.rs` — corpus ingest + query replay + `recall@k` / `nDCG@10`
  computation. Pure metric functions are unit-testable without a model.
- `benches/bench_eval.rs` — criterion bench that runs the harness end-to-end
  and emits `RECALL_AT_10=<float>` and `NDCG_AT_10=<float>` on stderr for CI
  scraping.
- `tests/fixtures/eval_baseline.toml` — committed reference scores
  (`recall@10`, `nDCG@10`) the CI gate compares against.
- Optional real-corpus slot: if the env var `ANNO_RAG_EVAL_CORPUS` is set, the
  harness reads `{corpus/, queries.toml}` from that path instead of the
  versioned fixtures. Real documents are never committed.

### Component B — Hybrid retrieval

Location: `crates/anno-rag/src/{embed,store,pipeline}.rs`

- `embed.rs`:
  - `embed_batch(passages)` keeps the `"passage: "` prefix (ingest path,
    unchanged).
  - New `embed_query(&str) -> Result<Vec<f32>>` applies the `"query: "`
    prefix. Same forward pass, pooling, and L2 normalization.
- `store.rs`:
  - After ingest, build a native LanceDB FTS (Tantivy) index on the
    `text_pseudo` column, alongside the existing vector index. Idempotent,
    rebuilt if absent — mirrors `maybe_build_index`.
  - `search()` becomes hybrid:
    `tbl.query().nearest_to(query_vec).full_text_search(query_text)
        .rerank(RRFReranker::default()).limit(k)`.
    The `query_text` passed to FTS is the **pseudonymized** query text, to
    stay consistent with the indexed `text_pseudo`.
- `pipeline.rs`:
  - `search()` embeds the query via `embed_query`, passes the pseudonymized
    query text to FTS, and lets RRF fuse the two result sets.
  - `ingest_one()` stays on `embed_batch` — unchanged.

Hybrid is the **default** for all callers (CLI, MCP, bench). No opt-in flag.

### `SearchHit` API change

With RRF, LanceDB returns a `_relevance_score` (higher is better) rather than
the L2 `_distance` (lower is closer). `SearchHit` gains a `score: f32` field
(relevance, higher = better). The existing `distance` field is removed. This is
a minor breaking change; anno-rag is pre-1.0.

## 4. Eval corpus and query format

### Synthetic corpus (~60 docs, versioned)

Document families, ~10 each, 1–3 KB apiece:
contrats de prestation, CDI/CDD, mises en demeure, baux commerciaux, CGV,
accords de confidentialité, statuts de société.

Written in realistic French legal register: article numbers, standard clauses,
domain terminology ("le Prestataire", "résiliation de plein droit", "préavis"),
amounts, dates. Synthetic PII (fictional names, fake IBAN/SIRET) is scattered
through the text to exercise the pseudonymization path without GDPR exposure.

Thematic overlap between documents is **intentional** — multiple contracts
discuss résiliation, confidentialité, etc. Without overlap, recall saturates at
1.0 (the v0.5 failure mode).

### `queries.toml` format

```toml
[[query]]
text = "résiliation anticipée avec indemnité"
relevant = [
  { doc = "prestation_03.txt", grade = 3 },
  { doc = "cdi_07.txt",        grade = 2 },
  { doc = "bail_02.txt",       grade = 1 },
]
```

Relevance grades:
- `3` — directly addresses the query
- `2` — partially relevant
- `1` — marginal mention
- absent from `relevant` — not relevant (grade 0)

Queries are **concept-oriented** (legal notions), not named-entity lookups.
This is the realistic legal-RAG use case and sidesteps the deferred
pseudonymization-vs-retrieval tension (a pseudonymized `[PERSON_1]` in a query
does not match the differently-numbered `[PERSON_1]` in a document — a known
limitation, documented, addressed in a later spec).

## 5. Metrics

Computed in `src/eval.rs`, pure functions, unit-tested on toy inputs.

- **recall@k** (k = 10): fraction of grade ≥ 1 documents retrieved within the
  top-k results, per query, macro-averaged across the corpus.
- **nDCG@10**: DCG over the graded relevance scores, normalized by the ideal
  DCG, macro-averaged. Rewards correct *ranking*, which is what the RRF rerank
  is expected to improve.

## 6. CI gates

In `bench.yml`:

- `bench_eval` runs on the versioned synthetic corpus (reproducible; the real
  corpus slot is never used in CI).
- Hard gate: **recall@10 ≥ baseline × 0.98** AND **nDCG@10 ≥ baseline × 0.98**.
  The 0.98 factor absorbs measurement noise.
- The v0.5 `bench_recall` (5-doc) is **removed** — superseded by `bench_eval`.
  `bench_search`, `bench_ingest`, `bench_cold_start`, `bench_idle_rss` remain.
- `eval_baseline.toml` is committed. Raising the baseline is an explicit,
  reviewed change in the PR that earns the improvement.

## 7. Testing

- `eval.rs`: unit tests for `recall@k` and `nDCG@10` on toy inputs with known
  expected values — no model required.
- `queries.toml`: a consistency test — every referenced `doc` exists in the
  corpus directory, every `grade` is in 0–3.
- `embed.rs`: a test that `embed_query` and `embed_batch` apply different
  prefixes (verifiable by inspection without HF, or an `#[ignore]`d test with a
  warmed model).
- `store.rs`: a hybrid-search test on a small set — a document that matches
  lexically but not semantically must surface via the FTS arm.
- End-to-end: `bench_eval` itself is the integration test.

## 8. Implementation risks

- **LanceDB 0.27.2 hybrid API maturity.** The implementation plan starts with a
  spike validating `full_text_search` + `RRFReranker` on LanceDB 0.27.2. If the
  native hybrid API is not usable, the documented fallback is two separate
  queries (vector + BM25) fused with a hand-written RRF — same result, more
  code. This decision is made early in implementation, not in this spec.

## 9. Migration notes

- `SearchHit.distance` is replaced by `SearchHit.score` (higher = better).
  Minor breaking API change; anno-rag is pre-1.0.
- A v0.6 entry is added to `crates/anno-rag/CHANGELOG.md`.
