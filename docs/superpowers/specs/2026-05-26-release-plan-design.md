# Anno Release Plan Design

Date: 2026-05-26

## Goal

Produce OS-specific optimized releases that can be tested in Cowork/Claude Desktop without using local debug binaries. The immediate release objective is an internal release candidate that replaces `D:\cargo-shared-target\debug\anno-rag.exe` with a GitHub-built release artifact or `.mcpb` package, then uses real logs and timings to validate performance.

## Current Context

- `main` is green on GitHub Actions at commit `20ff8f692d79377ea55902eb04f68a79a18f3897`.
- `Cargo.toml` already configures `cargo-dist` for GitHub releases.
- Release targets are Windows MSVC, macOS Apple Silicon, macOS Intel, and Linux x64.
- Installers already include MSI, PowerShell, shell, and Homebrew.
- The release workflow already packages a `.mcpb` extension with `anno-rag` embedded under `server/`.
- Git tags exist up to `v0.10.0`, but no GitHub Releases are currently published.
- Claude Desktop currently launches `D:\cargo-shared-target\debug\anno-rag.exe mcp`, which is unsuitable for performance testing.

## Release Channels

### RC Channel

The RC channel is the first target. It is for Cowork and local integration testing only.

An RC release is acceptable when:

- `main` CI is green.
- The release workflow completes successfully for all supported OS targets.
- Windows `.mcpb` and Windows release binary assets are produced.
- Claude/Cowork can run `anno-rag` from a release artifact instead of a debug target.
- `anno-rag mcp` starts and `anno_health` responds.
- Basic ingest/search timing is captured from logs.

### Stable Channel

Stable releases are promoted only after one or more RCs have been tested in Cowork.

A stable release is acceptable when:

- The RC has no blocking startup, packaging, or performance regressions.
- Installation instructions are documented.
- Rollback is documented.
- Release notes identify user-visible changes and known limitations.

## Versioning

Use SemVer tags that match a dist-able package version.

Because the workspace currently has `v0.10.0` tags already, the next RC should use a new version. The selected first RC version is `0.11.0-rc.1`, because the Legal Graph work is user-visible and should be validated as part of the release.

1. Bump the relevant release version to `0.11.0-rc.1`.
2. Tag the same version as `v0.11.0-rc.1`.
3. Push the tag to trigger `.github/workflows/release.yml`.

If `cargo-dist` rejects a prerelease version for the selected package layout, use `0.11.0` for the package version and tag `v0.11.0-rc.1` only if `dist plan` confirms the mapping. Do not push a tag until `dist plan` accepts the version locally or in a dry-run workflow.

## Artifact Requirements

Each RC must produce:

- Windows optimized `anno-rag.exe`.
- Windows `.mcpb` package.
- macOS ARM64 optimized `anno-rag`.
- macOS x64 optimized `anno-rag`.
- Linux x64 optimized `anno-rag`.
- Installer or archive assets generated by `cargo-dist`.
- Checksums generated by `cargo-dist`.

The `.mcpb` manifest must point to the embedded release binary:

- Windows: `${__dirname}/server/anno-rag.exe`
- Unix-like platforms: `${__dirname}/server/anno-rag`

## Cowork Test Flow

1. Download the Windows RC asset from the GitHub prerelease.
2. Install the `.mcpb` package when possible.
3. If `.mcpb` installation is not available, configure Claude Desktop to point to the extracted release binary.
4. Stop existing debug MCP processes.
5. Restart Claude/Cowork.
6. Confirm no active `anno-rag.exe` process runs from `D:\cargo-shared-target\debug`.
7. Run the normal Cowork interaction that was previously slow.
8. Capture logs, startup time, first query latency, ingest latency, CPU, and memory.

## Validation Checklist

### Before Tagging

- `git status` is clean.
- `origin/main` equals local `main`.
- Latest CI on `main` is successful.
- Release version and tag are coherent.

### After Release Workflow

- Release workflow is successful.
- GitHub Release exists.
- Expected assets exist.
- `.mcpb` exists for Windows at minimum.
- The `.mcpb` manifest uses the embedded binary path.
- Windows asset is built in release mode.

### Cowork Runtime

- Claude/Cowork no longer points to `target/debug`.
- No running MCP process uses `D:\cargo-shared-target\debug\anno-rag.exe`.
- `anno_health` responds.
- Logs show the expected release binary path.
- Performance timings are captured before and after the switch.

## Rollback

Rollback is configuration-based:

1. Stop the release MCP process.
2. Restore the previous Claude Desktop MCP config if needed.
3. Restart Claude/Cowork.
4. Record the failure reason in the RC notes.

Rollback should not require deleting release assets or rewriting tags. If an RC is bad, publish a new RC tag.

## Pipeline Hardening After First RC

After the first RC proves the release route works, add automated release checks:

- Smoke test `anno-rag mcp` on each OS.
- Smoke test `anno_health` through the MCP server.
- Validate `.mcpb` manifest command and entry point.
- Validate release artifacts are not debug builds.
- Upload a concise release verification report.
- Add documentation for installing the `.mcpb` in Claude Desktop/Cowork.

## Non-Goals

- Do not rely on local release builds as the primary distribution path.
- Do not publish a stable release until Cowork has validated the RC.
- Do not change the runtime model loading behavior as part of the release plan unless a release smoke test exposes a packaging bug.
- Do not delete existing tags.

## Selected Defaults

- Publish the first RC as a GitHub prerelease, not workflow artifacts only.
- Require Windows `.mcpb` for the first RC because Cowork testing is currently on Windows.
- Treat macOS and Linux `.mcpb` packaging as best-effort in the first RC, but require optimized binaries for all configured `cargo-dist` targets.
- Promote to stable only after the Windows Cowork test confirms release-mode performance and no startup regression.
